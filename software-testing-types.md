# 애플리케이션 테스트 종류 정리

출처: https://edder773.tistory.com/198

## 1. 애플리케이션 테스트 기본 개념

- 애플리케이션에 잠재된 결함을 찾아내는 일련의 행위 또는 절차다.
- 완벽한 테스트는 불가능하다.
- 파레토 법칙: 20%의 코드에서 80%의 결함이 발견된다.
- 살충제 패러독스: 같은 테스트를 반복하면 새로운 결함을 찾아내지 못한다.
- 테스팅은 정황(맥락)에 의존한다.
- 오류-부재의 궤변: 결함을 모두 제거해도 사용자 요구사항을 충족하지 못하면 품질 좋은 소프트웨어가 아니다.
- 테스트와 위험은 반비례 관계다.
- 테스트는 점진적으로 확대해 나간다.
- 테스트는 별도의 팀이 수행하는 것이 바람직하다.

## 2. 실행 여부에 따른 분류

| 구분 | 설명 |
|---|---|
| 정적 테스트 | 프로그램을 실행하지 않고 명세서·소스 코드를 분석한다. 워크스루, 인스펙션, 코드 검사 등이 있다. |
| 동적 테스트 | 프로그램을 실행해 오류를 찾는다. 블랙박스 테스트, 화이트박스 테스트로 나뉜다. |

## 3. 테스트 기반에 따른 분류

| 구분 | 설명 |
|---|---|
| 명세 기반 테스트 | 사용자 요구사항 명세를 빠짐없이 테스트한다. 동등 분할, 경계 값 분석 등을 사용한다. |
| 구조 기반 테스트 | 내부 논리 흐름에 따라 테스트 케이스를 작성한다. 구문 기반, 결정 기반, 조건 기반 등이 있다. |
| 경험 기반 테스트 | 테스터의 경험을 기반으로 수행한다. 에러 추정, 체크리스트, 탐색적 테스트 등이 있다. |

## 4. 시각에 따른 분류

| 구분 | 설명 |
|---|---|
| 검증 테스트 (Verification) | 개발자 시각에서 생산 과정을 테스트한다. |
| 확인 테스트 (Validation) | 사용자 시각에서 결과를 테스트한다. |

## 5. 목적에 따른 분류

| 테스트 유형 | 설명 |
|---|---|
| 회복 테스트 | 결함을 주입한 뒤 올바르게 복구되는지 확인한다. |
| 안전 테스트 | 불법적인 침입으로부터 시스템을 보호할 수 있는지 확인한다. |
| 강도 테스트 | 과부하 상황에서도 정상적으로 실행되는지 확인한다. |
| 성능 테스트 | 응답 시간, 처리량 등을 진단한다. |
| 구조 테스트 | 내부 논리 경로, 복잡도 등을 평가한다. |
| 회귀 테스트 | 변경된 코드가 새로운 결함을 발생시켰는지 확인한다. |
| 병행 테스트 | 변경 전후 소프트웨어의 결과를 비교한다. |

## 6. 화이트박스 테스트

원시 코드의 모든 논리적 경로를 테스트하는 방식이다.

**검증 기준**
- 문장 검증 기준
- 분기 검증 기준
- 조건 검증 기준
- 분기/조건 기준

**종류**
- 기초 경로 검사
- 제어 구조 검사 (조건 검사, 루프 검사, 데이터 흐름 검사)

## 7. 블랙박스 테스트

각 기능이 완전하게 작동하는지를 입증하는 방식이다.

**종류**
- 동치 분할 검사
- 경계값 분석
- 원인-효과 그래프 검사
- 오류 예측 검사
- 비교 검사

## 8. 통합 테스트

| 방식 | 설명 |
|---|---|
| 빅뱅 통합 테스트 | 비점진적 방식. 모든 모듈을 한 번에 결합해 테스트한다. |
| 하향식 통합 테스트 | 상위 모듈에서 하위 모듈 방향으로 통합하며, 하위 모듈 대신 스텁(Stub)을 사용한다. |
| 상향식 통합 테스트 | 하위 모듈에서 상위 모듈 방향으로 통합하며, 상위 모듈 대신 드라이버(Driver)를 사용한다. |
| 혼합식 통합 테스트 | 상향식과 하향식을 혼합한 방식으로, 샌드위치식이라고도 한다. |

## 9. 회귀 테스팅

변경된 모듈이 새로운 오류를 발생시켰는지 확인하기 위해 반복적으로 수행하는 테스트다.

## 10. 애플리케이션 테스트 프로세스

1. 테스트 계획
2. 테스트 분석 및 디자인
3. 테스트 케이스 및 시나리오 작성
4. 테스트 수행
5. 테스트 결과 평가 및 리포팅
6. 결함 추적 및 관리

## 11. 테스트 케이스 / 시나리오 / 오라클

| 용어 | 설명 |
|---|---|
| 테스트 케이스 | 요구사항 준수 여부를 확인하기 위한 입력값, 실행 조건, 예상 결과 등을 정리한 명세서 |
| 테스트 시나리오 | 여러 테스트 케이스를 적용하는 순서를 정리한 문서 |
| 테스트 오라클 | 테스트 결과가 참인지 판단하기 위해 기대 값과 비교하는 기준 |

**테스트 오라클 종류**
- 참(True) 오라클
- 샘플링(Sampling) 오라클
- 추정(Heuristic) 오라클
- 일관성 검사(Consistent) 오라클

## 12. 결함 관리

**결함 관리 프로세스**
계획 → 기록 → 검토 → 수정 → 재확인 → 추적/모니터링 → 분석/보고

**결함 추적 순서**
등록 → 검토 → 할당 → 수정 → 보류 → 종료 → 해제

**결함 분류**
시스템 결함, 기능 결함, GUI 결함, 문서 결함

**결함 관리 도구**
Mantis, Trac, Redmine, Bugzilla

## 13. 성능 측정 지표

- 처리량
- 응답 시간
- 경과 시간
- 자원 사용률

## 14. 테스트 자동화 도구

| 종류 | 설명 |
|---|---|
| 정적 분석 도구 | 코드를 실행하지 않고 결함을 분석한다. |
| 테스트 실행 도구 | 데이터 주도 방식, 키워드 주도 방식이 있다. |
| 성능 테스트 도구 | JMeter, LoadUI, OpenSTA 등 |
| 테스트 통제 도구 | 테스트 계획 및 관리를 지원한다. |
| 테스트 하네스 도구 | 테스트 환경(스텁, 드라이버 등)을 지원한다. |
