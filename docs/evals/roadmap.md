# Eval 로드맵 (일반 참고 문서, week 무관)

Anthropic의 "Evaluating agents" 로드맵을 이 프로젝트용으로 정리한 문서. 특정 week에 종속된 내용은 없다 — week별 구체 task 목록은 `week{N}-tasks.md`에 따로 둔다 (`README.md`에 인덱스).

핵심 원칙: **성공 기준을 조기에 명확히 정의하고, 꾸준히 측정하며, 지속적으로 개선한다.**

## Step 0. 빨리 시작하기

Task가 정의돼야 평가를 시작할 수 있다. 처음부터 완벽한 세트를 만들려다 미루는 게 제일 흔한 실수 — 초기엔 20~50개짜리 간단한 task만으로 충분하다. 성숙한 agent는 미세한 변화를 잡아내려고 더 크고 어려운 평가가 필요해지지만, 시작 단계에선 80/20이 최선이다.

**이 프로젝트에 적용하면**: 지금 이 repo엔 eval이 전혀 없었다. 미루지 말고 지금 있는 Week3 agent 기준으로 바로 20~30개 task를 만든다 (`week3-tasks.md`).

## Step 1. 수동 테스트부터 자동화하기

이미 개발하면서 반복적으로 손으로 치고 있는 질문들이 있을 것이다. 이런 것부터 평가 세트에 넣는 게 제일 싸고 확실하다. 서비스 중인 agent라면 버그 리포트/VOC를 task로 변환하는 것도 좋다.

**이 프로젝트에 적용하면**: `docs/troubleshooting/week*.md`에 이미 "증상/원인/해결" 형태로 기록된 회귀 사례들과, 디버깅하면서 실제로 Gradio 채팅창에 쳐본 문장들이 1차 소스다. 새로 지어낸 문장보다 이미 검증된 실패/성공 사례를 task로 옮기는 걸 우선한다.

## Step 2. Task가 명확한지 확인하기

좋은 task는 생각보다 만들기 어렵다. 기준 두 가지:
1. 전문가 2명에게 독립적으로 pass/fail 판단시켰을 때 결과가 갈리면, 그 task는 해석 여지가 있다는 뜻 — 노이즈만 추가하니 보완 필요.
2. Agent가 의도한 절차를 완벽히 따랐는데도 fail이면, task 자체가 풀 수 없거나 grader 설정이 잘못된 것. 이럴 땐 실제로 통과 가능한 레퍼런스 정답을 만들어서 task/grader 둘 다 검증한다.

**이 프로젝트에 적용하면**: kind 분류처럼 경계가 애매한 task(멤버 1명일 때 group인지 personal인지)는 "기대 결과"를 애매하게 두지 말고, 이번에 합의한 정책(멤버 있으면 무조건 group_schedule)을 명시적으로 박아서 판단 기준을 하나로 고정한다.

## Step 3. 균형 잡힌(balanced) 테스트 셋 만들기

"해야 하는 걸 했는지"만큼 "하면 안 되는 걸 안 했는지"도 중요하다. Tool 호출 여부만 평가하면 모든 요청에 무조건 tool을 호출하는 agent가 만들어진다. Undertriggering과 overtriggering 균형을 계속 검토하면서 task를 추가해야 한다. (Anthropic 자체도 웹 검색 커버리지 균형 잡는 데 계속 고생 중이라고 함.)

**이 프로젝트에 적용하면**: "일정 저장해줘" 류의 golden task만 쌓지 말고, "안녕", "오늘 날씨 어때" 같이 저장 tool을 호출하면 안 되는 negative task를 반드시 같이 둔다.

## Step 4. 안정적인 평가 하네스 구축하기

Task가 아무리 좋아도 하네스가 부실하면 신뢰 못 한다. 지켜야 할 것 두 가지:
- 모든 task는 서로 영향 없이 독립적으로 실행돼야 한다.
- Agent 외적인 요인(인프라 제약, 코드 버그)으로 실패하면 안 된다.

**이 프로젝트에 적용하면**: 지금은 하네스 자체가 없다 (정적 문서만). 나중에 자동화 스크립트를 만들 때, `fixed/app_store.py`가 쓰는 SQLite 파일(`data/*.db`)을 매 task 실행 전에 초기화하지 않으면 이전 task가 저장한 row가 남아서 "일정 몇 개 저장됐는지" 같은 task가 서로 오염된다 — 반드시 task마다 격리된 DB 파일/트랜잭션 롤백을 쓸 것.

## Step 5. Grader를 정성 들여 설계하기

원칙:
- 가능하면 deterministic grader (동일 출력엔 항상 동일 판정).
- 유연한 평가가 꼭 필요할 때만 AI 모델 grader.
- AI 모델 grader는 사람이 검증.
- **Agent의 흐름(tool 호출 순서)보다 output(최종 결과)에 집중.** 순서를 강제하면 변경에 취약해지고, agent가 찾아낸 대안 경로를 전부 놓친다.

**이 프로젝트에 적용하면**: "일정 알려줘" task를 채점할 때 "list_saved_requests를 호출했는지"보다 "응답에 개인/그룹/할 일/리마인더가 다 언급됐는지" 같은 최종 결과를 본다. Tool을 어떤 순서로 불렀든 최종 답변/DB 상태만 맞으면 통과.

### 부분 점수

Task가 여러 단계면 단계별 부분 점수를 준다. 고객 식별엔 성공했지만 환불 처리에 실패한 agent는 처음부터 실패한 agent보다 낫다 — 연속적인 점수로 평가.

**이 프로젝트에 적용하면**: "요청 구조화 → SQLite 저장 → 조회 반영"처럼 3단계짜리 task는 셋 중 몇 단계까지 성공했는지로 채점 (예: 구조화까진 맞았는데 kind가 틀렸다 = 1/3).

### LLM grader 검토

LLM grader는 사람 전문가가 캘리브레이션해야 하고, LLM grader도 할루시네이션 낼 수 있다. 정보 부족하면 "Unknown"을 내도록 가이드하고, 여러 측면을 한 번에 평가하지 말고 측면별로 별개 LLM에게 맡긴다. 초기엔 공수가 크지만, 한 번 견고해지면 이후엔 가끔만 확인하면 된다.

### 치팅 방지

Agent가 평가를 우회/치팅하기 쉬우면 안 된다.

## Step 6. Transcript(테스트 결과) 확인하기

Grader가 잘 동작하는지는 transcript를 실제로 봐야 안다. 실패한 결과는 "agent가 진짜 실수한 건지 vs grader가 잘못 판정한 건지"가 명확해야 한다. 점수가 안 오를 때 그게 "평가가 잘못된 것"이 아니라 "agent 성능 문제"라는 확신이 있어야 한다.

**이 프로젝트에 적용하면**: Gradio "상세" 탭의 trace JSON이 이미 이 역할을 한다 — task가 fail이면 trace에서 어느 tool 호출이 잘못됐는지 먼저 확인하고, task/grader 문제인지 agent 문제인지부터 구분한다.

## Step 7. 평가 능력 포화(Eval Capability Saturation) 모니터링

모든 task를 100% 통과하는 평가 스위트는 회귀는 잡아주지만 개선 신호는 안 준다("졸업"한 상태). 포화될수록 점수 상승이 느려지고 가장 어려운 문제만 남아서, 실제로는 성능이 크게 늘었는데 점수는 미미하게만 오른 것처럼 보이는 착시가 생긴다. 표면 점수만 보지 말고 세부 결과를 계속 검토해야 하고, 쉬운 task 비중을 줄이고 어려운 task를 늘리는 것도 방법이다.

**이 프로젝트에 적용하면**: `week3-tasks.md`의 golden/boundary task가 전부 통과하기 시작하면, "더 애매한 경계 사례"나 "여러 kind가 한 문장에 섞인 요청" 같은 더 어려운 task를 추가해서 변별력을 유지한다.

## Step 8. 공개 기여/유지보수를 통해 최신화하기

평가 스위트는 살아있는 아티팩트다. 지속적 관심과 명확한 책임자가 있어야 유지된다. 평가에 책임을 지고 task를 정의하는 것 자체가 "요구사항이 개발 시작하기에 충분히 구체적인지" 검증하는 방법이다. Anthropic은 이를 **평가 주도 개발(Eval-driven Development)** 이라 부른다 — 기능을 만들기 전에 그 기능을 정의하는 평가부터 만들고, agent가 통과할 때까지 반복.

**이 프로젝트에 적용하면**: 확장 전략은 `README.md`(인덱스)와 `week{N}-tasks.md` 명명 규칙으로 관리한다 — 새 주차 시작할 때 새 task 파일부터 만들고, 그 주차 구현을 그 task들이 통과하도록 진행하는 순서를 권장.
