# 3주차 학습 노트 (개인 정리)

> `docs/week03.md`(공식 커리큘럼 문서)와 별개로, 수강 중 남긴 메모를 맥락 보충해서 정리한 개인 노트.

## 1. 2주차 복습 — Transformer, 구조화된 출력

- 2주차에서 LLM 구조(Transformer)를 설명 들었지만, "그냥 그렇게 생겼구나" 정도로 넘어간 상태 — 완전히 소화된 개념은 아님. 예전 실습에서도 temperature·뉴런 수에 따른 결과 차이 정도만 관찰했지, 내부 동작 원리까지 파고들진 않았음.
- 2주차의 핵심 결과물은 **구조화된 출력(structured output)** 구현이었음 — 자유 텍스트 응답 대신 `BaseModel` 기반 스키마에 맞춰 모델이 값을 채우도록 만드는 것.

## 2. 3주차 핵심 — 구조화된 출력 → 내부 시스템 연계

3주차는 "구조화된 출력이 나온 다음, 그걸 실제 프로그램·저장(DB)과 어떻게 연결하는가"를 다룸.

- 노트북 실습: `SaveStructuredRequestInput(BaseModel)`을 tool의 **입력 스키마(`args_schema`)**로 지정 → 모델이 자연어 요청을 이 스키마에 맞는 arguments로 변환 → `save_structured_request` tool이 이를 받아 SQLite에 저장.
- 관건은 **"tool에 필요한 데이터를 얼마나 잘 전달하느냐"**. 즉 모델이 tool을 호출할 때 넘기는 arguments의 정확도/완결성이 전체 파이프라인의 성패를 좌우함.
- 필드에 `description`을 다는 것도 도움이 되지만, 실제 저장 단계에서는 **타입(type)**이 더 크게 작용한다 — 예: `Literal["personal_schedule", "group_schedule", "todo", "reminder", "unknown"]`처럼 타입으로 값의 범위를 강제하면, 설명 문구보다 훨씬 확실하게 데이터 형태를 통제할 수 있음.

## 3. DB 구성과 에이전트의 관계

- 맨 처음 DB(테이블) 구성 자체는 에이전트와 직접 상관없는 부분 — table을 만들고 "필요한 정보만" 저장하도록 설계하는 건 일반 백엔드 설계의 영역.
- LangChain에서 DB를 다루는 파트로 **memory-saver(checkpoint)**가 있음. 이건 대화 이력을 저장하는 용도로, 에이전트 쪽(`create_agent`)에 구현되는 파트. 즉:
  - **대화 자체(메시지 히스토리)**는 agent의 `invoke` 안에서 자동으로 저장/관리됨 (checkpoint 메커니즘).
  - 반면 **구조화된 요청 데이터**(일정/할일/알림 등 실제 비즈니스 데이터)는 별도로 tool을 통해 추출 → DB에 연결 → `execute`로 insert하는 흐름 — 노트북의 `save_structured_request` tool이 이 역할.
- 즉 "대화 저장"과 "데이터 저장"은 서로 다른 메커니즘이라는 점이 핵심 구분.

## 4. Tool 설계 시 주의점

- 상용 프로그램을 agent 형태로 만들 때, tool을 실행하는 work 함수는 인자(LLM이 넘기는 값)를 **프롬프트로 제대로 제어해주지 않으면 오작동할 수 있음** — 즉 시스템 프롬프트에서 분류 기준·날짜 변환 규칙 등을 명시해야 tool arguments가 안정적으로 채워짐 (노트북의 system_prompt에서 "오늘은 2026-05-15", "상대 날짜는 YYYY-MM-DD로 변환" 등을 명시한 것과 동일한 이유).
- `@tool` 데코레이터에서 `args_schema` 파라미터로 입력 스키마를 고정할 수 있음 (`args_schema=SaveStructuredRequestInput`) — 이렇게 하면 tool 함수는 `**kwargs`로 받되, Pydantic이 스키마 검증을 강제해줌.
- **tool call이 실패해도 에이전트가 즉시 멈추지 않고 계속 재시도/다음 액션으로 넘어가는 구조**이기 때문에, 이 과정에서 실패가 조용히 묻히는 **loss**가 발생할 수 있음 — 실패를 명시적으로 잡아내는 로직이 없으면 눈치채기 어려움.
- 결론적으로 **에이전트 전체 성능은 결국 tool call의 품질에 달려있음** — arguments가 바뀌는 부분(스키마 매핑, 타입 캐스팅, 값 누락 등)을 계속 체크해야 함.

## 5. 생성 모델의 검증 필요성

- 생성 모델은 정밀한 제어가 어려운 특성상, **"모델을 구현했다"로 끝내면 안 되고 반드시 검증 단계가 필요**.
- 에이전트를 평가하려면 **평가용 dataset을 직접 만들어야 함** (기성 벤치마크가 없는 경우가 많으므로).
  - 방법 1: 더미 데이터를 만들거나, 프롬프트만으로 케이스를 검사.
- 생성 모델 검증 방식 3가지:
  1. **사람이 직접 확인 (gold standard)** — 가장 신뢰도 높지만 비용/시간 소모.
  2. **전통적 딥러닝 방식** — 질문-정답 쌍(ground truth)을 미리 만들어두고, 생성된 결과와 비교(정확도/일치도 측정).
  3. **LLM을 평가자로 사용 (LLM-as-judge)** — 다른 LLM이 생성 결과를 채점.

## 6. 3주차 과제 흐름

- 1주차: 임시로 저장(잠깐 메모리에 들고 있던 것)
- 2주차: 구조화된 출력으로 변환
- 3주차: 그 출력을 **검증한 뒤 DB에 저장**
- 이 전체 흐름에서 tool argument를 얼마나 잘 활용하느냐가 핵심 — LangChain의 구조화된 출력 기능이 이 파이프라인의 중심.

## 7. 기타 개념 메모 (다음 주 예고성 내용)

- **Lost in the middle**: context가 길어질 때 모델이 프롬프트/문서의 **중간 부분 정보를 놓치는 문제** — context 길이 관리 시 유의해야 할 현상.
- **하네스(harness)**: 에이전트를 제어하는 방법론 — 우리가 직접 짤 수 있는 워크플로우 구성 방식을 의미 (도구 호출 순서, 검증 단계, 재시도 로직 등을 설계하는 틀).
- LangChain에서 **Deep Agent**라는 툴이 새로 나왔다는 언급 — 다음에 살펴볼 대상으로 메모해둔 것으로 보임.
