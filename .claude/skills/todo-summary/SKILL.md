# todo-summary

과제 파일 하나를 분석해서 안에 있는 TODO들을 구조화해서 보여주는 스킬입니다.
목표는 "이 파일에서 뭘 채워야 하는지"를 함수 단위로 한눈에 보이게 정리하는 것입니다.

## 입력

- 인자로 과제 파일 경로 하나를 받습니다 (예: `student_parts/week03_build_nanas_logbook.py`).
- 경로가 없으면 `student_parts/*.py` 목록을 보여주고 어떤 파일을 볼지 되묻습니다.
- 경로는 있는데 파일이 존재하지 않거나 읽을 수 없으면, TODO 파싱으로 우회하지 않고 그 사실을
  바로 사용자에게 알립니다(파싱 실패가 아니라 입력 문제이므로 아래 fallback과 구분).

## 분석 절차

파일마다 헤딩 문구·섹션 개수·tier 표기 유무가 다르므로, 특정 헤딩 이름을 하드코딩하지
말고 아래 순서로 구조를 스스로 찾아냅니다.

### 1단계 — 가이드 블록 범위 찾기

- 시작: 정규식 `#\s*\[.*수강생\s*구현\s*가이드\]` 에 매칭되는 라인.
  (예: `# [수강생 구현 가이드]`, `# [2주차 1회차 수강생 구현 가이드]`, `# [3주차 수강생 구현 가이드]`
  처럼 번호·회차 표기는 파일마다 다르므로 느슨하게 매칭)
- 끝: 이 라인부터 연속되는 `#` 주석 라인이 끝나고 실제 코드(`def`/`class`, 최상위 들여쓰기)가
  다시 시작되는 지점 바로 앞까지.

### 2단계 — "함수별 동작 설명" 섹션만 앵커로 사용

- 가이드 블록 안의 다른 소제목(목표/구현 대상/반환 규칙/검증 방법 등)은 파일마다 이름과
  개수가 다르니 파싱 대상으로 삼지 않습니다.
- `함수별 동작 설명` 문자열이 나오는 라인부터 가이드 블록 끝까지만 bullet 파싱 대상으로 삼습니다.
- 이 헤딩은 지금까지(week01~03)는 항상 존재하고 항상 마지막이었지만, 새 주차 파일에서도
  그렇다고 가정하지 않습니다. 매번 현재 파일에서 실제로 찾습니다.
- **앵커 이름이 다를 때**: "함수별 동작 설명"과 뜻이 같은 다른 표현(예: "함수 설명",
  "구현 대상별 설명" 등)이 있는지 느슨하게 재탐색합니다.
- **앵커는 있는데 마지막이 아닐 때**: 이 헤딩부터 다음 `#제목` 라인(정규식 `^#\s*[가-힣]`으로
  시작하는 새 헤딩) 또는 가이드 블록 끝 중 먼저 오는 지점까지로 범위를 잡습니다.
- **구조 파싱 실패 (앵커를 끝내 못 찾았거나, 찾았는데 3단계 bullet 정규식이 그 범위 안에서
  0건 매칭일 때)**: 함수→설명 테이블 없이 4~6단계(TODO 수집·소속 함수 역추적·병합)만
  수행합니다. 이때 출력 맨 위에 반드시 원인을 한 줄로 명시합니다(조용히 생략하지 않습니다):
  - "이 파일엔 '함수별 동작 설명' 섹션이 없어 TODO만 나열합니다"
  - "가이드 섹션은 찾았지만 형식이 달라 함수 설명 매칭에 실패해 TODO만 나열합니다"

### 3단계 — bullet 항목 파싱해서 함수→설명 테이블 만들기

각 항목은 다음 형태입니다.

```
#   - [tier] 함수명(args)
#     설명 여러 줄...
#
```

- 정규식: `^#\s+-\s+(?:\[(메인|추가|공통)\]\s+)?(.+?)\s*$`
  - 그룹1 = tier (없으면 None — week01/week02는 tier 표기 자체가 없음, 이 경우 이후 출력에서 tier 접두어를 아예 생략)
  - 그룹2 = 이름 부분
- 이름 부분에 `/`가 있으면 분리합니다. 예: `week02_system_prompt() / week02_prompt_parts()`,
  `SavedRequestListInput / SavedRequestGetInput / ...` → 여러 함수/클래스가 같은 tier·설명을 공유합니다.
- 이름에서 `(...)` 괄호와 인자는 제거하고 순수 식별자만 남깁니다.
  `ClassName.method_name` 형태는 클래스와 메서드를 분리해서 기록합니다 (둘 다 같은 키로 조회 가능하게).
- 설명 텍스트는 다음 bullet 라인 또는 빈 `#` 줄이 나오기 전까지 이어지는 들여쓰기 라인들입니다.
- 결과를 `{함수명: {tier, desc}}` 형태의 테이블로 들고 있습니다.

### 4단계 — 파일 전체에서 TODO 라인 수집

- 정규식: `#\s*TODO:?\s*(.+)`
- 각 TODO 라인이 **어떤 함수/클래스 안에 있는지** 들여쓰기를 기준으로 역추적합니다.
  - 위로 올라가며 가장 가까운 `def NAME` 또는 `class NAME`을 찾으면 그 이름이 소속 함수입니다.
  - 그런 블록을 못 찾으면 (모듈 최상단, 어떤 def/class 안에도 없음) **모듈 레벨 TODO**로 분류합니다.
- 모듈 레벨 TODO는 3단계 테이블과 매칭하지 않고 그대로 독립 항목으로 둡니다.
  (보통 이미 한 줄로 충분히 설명되어 있음 — 억지로 다른 설명을 붙이지 않습니다.)
- 이 정규식으로 TODO가 0건이면, "이 파일엔 TODO가 없습니다"라고 명시적으로 답합니다.
  구조 파싱 실패(2단계)로 0건이 된 것과는 다른 상황이므로 서로 다른 문구로 구분해서 보여줍니다.

### 5단계 — 함수 내부 TODO를 테이블과 매칭

- 소속 함수명으로 3단계 테이블을 조회합니다.
- 있으면 tier + 설명을 붙입니다.
- 없으면 (가이드 설명에 아예 언급 안 된 함수 — 보통 "이후 회차 예약 함수"류) tier/설명 없이 TODO 원문만 보여줍니다.

### 6단계 — 같은 함수 안 연속 TODO 병합

- 같은 함수 안에 TODO가 여러 줄이면 (예: 라인 344, 345) 하나의 함수 그룹 아래 목록으로 묶습니다.
- 그룹 순서는 파일에 나오는 순서(라인 번호 오름차순)를 유지합니다.

## 출력 포맷

tier 정보가 하나도 없는 파일(week01, week02처럼)이면 tier 접두어를 통째로 생략합니다.
tier가 있는 파일(week03 이후)이면 붙입니다.

```
[모듈 레벨]
- TODO (line N): <TODO 원문 그대로 또는 자연스럽게 다듬은 한 줄>

[tier] 함수명 또는 클래스명 (line N~M)
  가이드: <함수별 동작 설명에서 가져온 요약, 테이블에 없으면 이 줄 생략>
  - TODO (line N): <내용>
  - TODO (line M): <내용>
```

- tier가 없는 파일이면 `[tier]` 부분을 통째로 빼고 `함수명 (line N~M)`만 씁니다.
- "가이드:" 줄은 테이블에 매칭된 경우에만 넣습니다. 억지로 만들어내지 않습니다.
- 함수 그룹은 파일에 등장하는 순서대로 나열합니다.

## 예시 (참고용 포맷 샘플 — 매번 재계산)

아래는 week03_build_nanas_logbook.py를 과거 한 시점에 분석했던 결과의 포맷 샘플입니다.
**라인 번호·TODO 내용·함수 목록은 파일이 수정되면 바뀌므로 절대 그대로 재사용하지 않습니다.**
이 스킬이 호출될 때마다 인자로 받은 파일을 처음부터 다시 읽고 위 1~6단계를 전부 재실행해서
값을 새로 계산합니다. 아래 예시는 오직 "출력 형식이 이렇게 생겼다"는 포맷 참고용입니다.


## 주의사항

- 헤딩 이름(목표/과제 구성/반환 규칙 등)을 하드코딩해서 파싱하지 않습니다. 파일마다 다르고
  개수도 다릅니다. "함수별 동작 설명" 섹션도 지금까지는 항상 존재했을 뿐, 새 주차 파일에서
  보장된 것은 아닙니다 — 2단계의 fallback 절차를 항상 거칩니다.
- 예시 섹션의 라인 번호·함수명·TODO 내용을 실제 값으로 재사용하지 않습니다. 항상 이번에
  받은 파일을 다시 읽어서 계산합니다.
- 가이드 블록 자체가 없는 파일도 2단계의 "구조 파싱 실패" fallback으로 처리합니다
  (모듈/함수 TODO만 나열 + 원인 한 줄 명시, 조용히 생략하지 않음).
- TODO 내용을 과하게 재작성하지 않습니다. 원문이 이미 충분히 구체적이면 그대로 쓰고,
  너무 축약된 경우에만 매칭된 가이드 설명을 참고해 보충합니다.
